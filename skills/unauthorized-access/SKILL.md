# 未授权访问检测（Missing Authentication）

## 目标
确认"应受保护"的接口在缺少认证凭证时，是否仍可读取敏感数据或执行受限操作。

## 适用场景
- 用户资料、订单、账单、管理后台、审批、配置变更等应受保护能力。
- 文档标注"需登录/需角色"但实际可能缺少后端鉴权的接口。

## 前置条件与安全边界
- 仅在授权环境执行验证，禁止越权改写生产数据。
- 单接口默认最多 6 次请求（基线 2 次 + 对照 2 次 + 复验 2 次）。
- 单轮只改变"是否携带认证信息"一个变量，不联动改参数。
- 一旦形成 confirmed 证据，立即停止扩展探测。

## 检测步骤
1. 确定目标接口是否应受保护（产品语义、文档、页面入口、业务字段）。
2. 使用有效登录态发送 1~2 次基线请求，记录状态码、响应体关键字段、业务副作用。
3. 删除全部认证信息（Cookie、Authorization、自定义 Token 头）后重放同请求。
4. 对比有/无凭证结果，重点验证是否仍返回敏感业务数据或执行成功。
5. 对可疑结论重复至少 1 轮复验，确认可稳定复现。

## 闭环验证要求（必须遵守）
- 所有结论必须形成"前置条件/输入 -> 系统处理 -> 实际效果/危害 -> 可复核证据"的完整证据链，重点证明风险真实生效，而不是只看响应。
- 仅凭上传成功、请求成功、状态码变化、单次报错、配置落盘、前端提示、登录跳转、返回文案等中间信号，不得直接判定 `confirmed`。
- 若只有中间信号、缺少真实效果/危害验证，结论最多为 `suspected`。
- 对写操作、上传、配置变更、认证命中、权限变更、解析触发等场景，必须补做访问、回读、触发、解析、执行或副作用验证，证明风险真正落地并说明具体危害。
- 无法闭环时，必须明确缺失环节与下一步验证建议，不得跳过验证直接下结论。

## 实际效果验证方向（至少证明一类）
- 无凭证情况下读取到本应受保护的真实数据。
- 无凭证情况下执行了本应受保护的写操作，并通过回读或状态变化证明生效。
- 若只有 200、跳转异常或页面轮廓，但未证明真实数据/能力可用，不能给 `confirmed`。

## 判定标准

| 现象 | 判定 |
|------|------|
| 无凭证仍可读取敏感数据或执行受限操作 | confirmed |
| 行为异常（如 200 空壳响应）但未证明数据/能力可用 | suspected |
| 无凭证返回 401/403/登录跳转且无敏感数据 | not vulnerable |

## 误报控制
- 不仅看状态码，必须看响应体中是否存在真实业务字段或成功副作用。
- 排除"公开接口"（登录、注册、公开信息查询）后再下结论。
- 对写操作必须做状态回读验证，确认变更确实生效。
- 若响应包含缓存/降级模板，需追加一次带随机查询参数的复验。

## 最小 PoC 输出模板
- 目标接口：`<METHOD> <URL>`
- 认证前提：`应为登录态/角色受限`
- 对照请求：
  - `Req-A（有凭证） -> <关键结果>`
  - `Req-B（无凭证） -> <关键结果>`
- 关键证据：`<敏感字段/成功副作用>`
- 结论：`confirmed | suspected | not vulnerable`

## 修复建议
- 在服务端强制执行统一鉴权中间件与资源级授权校验。
- 对"公开路由"与"受保护路由"做显式分组，禁止隐式放行。
- 对关键写操作增加审计日志，便于回溯未授权调用。
- 对历史接口做回归测试，确保无凭证访问基线一致返回未授权。
