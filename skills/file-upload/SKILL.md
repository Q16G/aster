# 文件上传检测：多策略综合（Orchestrator）

## 目标
在上传点类型和后端处理逻辑不明确时，先做上传面探测与基线采集，再按证据选择子 skill 深挖，最终输出可复现的最小 PoC 与修复建议。

## 使用方式（关键）
- 本技能是"路由器"：当用户只说"测文件上传""上传风险不确定""想要完整测试项"时优先使用。
- 执行时不要默认把所有子 skill 固定顺序串行跑完，而是按"通用流程 + 绕过技巧 + 效果验证"组合加载：
  - 先固定加载 `load_skill({"skill_key":"文件上传-通用测试流程与闭环验证"})`
  - 再按限制与异常选择 1 个或多个"绕过技巧 skill"：
    - `load_skill({"skill_key":"文件上传-扩展名与MIME绕过检测"})`
    - `load_skill({"skill_key":"文件上传-文件内容与解析绕过检测"})`
    - `load_skill({"skill_key":"文件上传-中间件配置文件上传检测"})`
  - 任一变体一旦被接收，或出现关键异常，再按真实落点加载"效果验证 skill"：
    - `load_skill({"skill_key":"文件上传-存储路径与访问控制检测"})`
    - `load_skill({"skill_key":"文件上传-恶意文件执行链检测"})`
- 最小组合是 2 个 skill：`文件上传-通用测试流程与闭环验证` + 至少 1 个绕过技巧 skill。
- 如果是跨维度变体，例如 `.user.ini + content + type`，允许同时加载多个绕过技巧 skill，但流程顺序、证据记录和 `confirmed` 判定仍由通用流程 skill 统一约束。

## 全局约束（默认保守预算）
- 单上传点总尝试不超过 25 次（含基线、失败重试）。
- 每轮只变更 1 个变量（扩展名、MIME、文件名、内容、路径参数之一），保证因果可解释。
- 避免直接上传高危可执行 payload 到生产；优先"无害标记文件 + 可观测行为"验证。
- 一旦确认高危链路（可执行或越权读取）即停止扩测，进入证据固化与修复建议阶段。

## 参数与面识别
优先收集以下信息：
- 上传入口：表单上传、分片上传、富文本上传、头像上传、导入接口。
- 约束信息：前端 accept、后端返回错误信息、文件大小限制、是否二次处理（压缩/转码/解压）。
- 文件落点：返回 URL、对象存储 Key、临时目录、CDN 地址。

## 统一证据记录（建议）
每次测试记录：
- 输入变更：文件名、扩展名、MIME、内容特征、请求参数
- 输出特征：status、错误码、响应关键字、返回地址、可访问性
- 访问验证：上传后 URL 是否可读、是否被当作可执行脚本处理
- 配置联动证据：配置文件上传响应 + 触发文件上传响应 + 触发访问结果
- 可复核性：采用足以证明结果稳定、有效、可靠的方式复核，不设统一固定次数

## 闭环验证原则（必须遵守）
- 上传成功、返回文件 URL、对象存储 key、文件名落盘都不等于漏洞成立。
- `confirmed` 必须包含至少一种后续验证：访问、下载回读、预览、解析、执行、跨身份访问或配置联动触发。
- 若只证明"能上传"，但没有证明"会被危险方式访问/解析/执行/越权读取"，结论最多为 `suspected`。
- 对写入型影响（覆盖、替换、配置改写），必须补做前后对比或回读验证，证明状态真实变化。

## 实际效果验证（必须回答）
- 文件最终会被谁、通过什么链路访问或解析：直接访问 URL、业务预览、导入解析、异步处理、下载回读，还是跨身份读取？
- 真实安全效果是什么：存储型 XSS / HTML 注入、脚本执行、模板执行、越权读取、覆盖写入、内容污染、资源消耗，还是仅仅"可上传"？
- 若无法明确说出"触发链 + 实际效果 + 可观察证据"，不得给 `confirmed`。

## 命中后测试规划（按角色拼接）
命中"文件上传"后，按以下角色组合执行，而不是固定把所有子 skill 都跑一遍：
1. **通用流程**
   - 加载 `文件上传-通用测试流程与闭环验证`。
   - 完成基线白名单上传、约束采集、文件落点识别、证据记录格式初始化。
2. **绕过技巧（按需选择 1 个或多个）**
   - `文件上传-扩展名与MIME绕过检测`：文件名、后缀、MIME、multipart part header 维度。
   - `文件上传-文件内容与解析绕过检测`：魔数、结构、polyglot、文本内容、解析输入维度。
   - `文件上传-中间件配置文件上传检测`：Apache 的 `.htaccess`、Nginx + PHP-FPM 常见的 `.user.ini`、IIS 的 `web.config` 与目录级规则改写。
3. **效果验证（至少选择 1 条与落点相符的通道）**
   - `文件上传-存储路径与访问控制检测`：URL 可访问、跨身份读取、覆盖写入、路径越界。
   - `文件上传-恶意文件执行链检测`：访问、预览、解析或异步处理后触发脚本/模板/执行链。
4. **复测与结论固化**
   - 对 `confirmed/suspected` 项做最小复现复测，输出 PoC 与修复建议。

## 组合示例（优先按证据选，不要全量串行）
| 场景 | 推荐组合 |
|------|----------|
| 仅出现"后缀不允许 / MIME 不匹配" | 通用流程 + `文件上传-扩展名与MIME绕过检测` + `文件上传-存储路径与访问控制检测` 或 `文件上传-恶意文件执行链检测` |
| 有缩略图、转码、OCR、导入解析 | 通用流程 + `文件上传-文件内容与解析绕过检测` + 对应效果验证 skill |
| 怀疑点号文件或目录级规则被读取 | 通用流程 + `文件上传-中间件配置文件上传检测` + `文件上传-恶意文件执行链检测` |
| `.user.ini + content + type` 这类跨维度变体 | 通用流程 + `文件上传-中间件配置文件上传检测` + `文件上传-扩展名与MIME绕过检测` + `文件上传-文件内容与解析绕过检测` + `文件上传-恶意文件执行链检测` |

## 测试项总清单（命中后应覆盖）
- 上传功能基线（白名单文件、大小限制、错误处理）
- 扩展名绕过（大小写、双扩展、尾随点/空格）
- MIME 绕过（伪造 Content-Type、multipart part MIME 不一致）
- 魔数与真实内容校验（扩展名与内容不一致）
- 解析处理链安全（转码、解压、导入、OCR、异步处理）
- 存储命名安全（可预测命名、同名覆盖、路径拼接）
- 文件访问控制（未登录访问、跨用户访问、租户隔离）
- 中间件配置文件上传（Apache 的 `.htaccess`、Nginx + PHP-FPM 常见的 `.user.ini`、IIS 的 `web.config`）
- 配置上传后的联动触发验证（配置成功不等于漏洞成立）
- 配置生效范围与解析规则改写风险（目录级配置影响面）
- 执行链风险（上传后访问/预览/处理触发脚本或模板执行）
- 审计与回归（复现稳定性、证据完整性、修复后回归）

## 决策树

### Step 0：基线与白名单探测
1. 上传合法白名单文件（如 `a.jpg`、`a.txt`），确认上传功能本身可用。
2. 记录成功响应结构、返回文件 URL 规则与落盘命名规则。

### Step 1：选择子技能
1. **固定先加载通用流程**
   - 动作：加载 `文件上传-通用测试流程与闭环验证`
2. **选择绕过技巧**
   - 线索：报错提示围绕"后缀/MIME 不允许"
   - 动作：加载 `文件上传-扩展名与MIME绕过检测`
   - 线索：上传后有异步处理、缩略图、解析结果，或怀疑存在内容层校验
   - 动作：加载 `文件上传-文件内容与解析绕过检测`
   - 线索：Apache、Nginx + PHP-FPM、IIS 环境；目录允许上传点规则文件（如 `.htaccess`、`.user.ini`、`web.config`）
   - 动作：加载 `文件上传-中间件配置文件上传检测`
   - 线索：变体跨多个维度，例如 `.user.ini + content + type`
   - 动作：允许同时加载多个绕过技巧 skill，不要求单 skill 独立完成全部变体
3. **选择效果验证**
   - 线索：返回可访问 URL、路径可猜测、命名规则可枚举
   - 动作：加载 `文件上传-存储路径与访问控制检测`
   - 线索：文件落在可执行目录、历史漏洞为 webshell/RCE、或配置改写后疑似可执行
   - 动作：加载 `文件上传-恶意文件执行链检测`

### Step 2：确认标准
- `confirmed`：必须证明"上传/变更 -> 访问/预览/解析/执行/回读 -> 具体安全效果"的完整闭环，与输入改动有明确因果，且证据足以说明结果有效可靠。
- 对中间件配置类风险，`confirmed` 必须包含"配置上传 + 触发文件 + 行为变化"的完整联动证据链。
- 仅上传成功、返回 URL 或配置落盘不算 `confirmed`，最多为 `suspected`。
- `suspected`：存在异常或中间信号，但未证明具体安全效果，或触发链仍缺关键一环。
- `not vulnerable`：在预算内未形成可复现风险证据。

## 闭环验证要求（必须遵守）
- 所有结论必须形成"前置条件/输入 -> 系统处理 -> 实际效果/危害 -> 可复核证据"的完整证据链，重点证明风险真实生效，而不是只看响应。
- 仅凭上传成功、请求成功、状态码变化、单次报错、配置落盘、前端提示、登录跳转、返回文案等中间信号，不得直接判定 `confirmed`。
- 若只有中间信号、缺少真实效果/危害验证，结论最多为 `suspected`。
- 对写操作、上传、配置变更、认证命中、权限变更、解析触发等场景，必须补做访问、回读、触发、解析、执行或副作用验证，证明风险真正落地并说明具体危害。
- 无法闭环时，必须明确缺失环节与下一步验证建议，不得跳过验证直接下结论。

## 最小 PoC 输出模板
- 目标：`<URL/接口>`
- 测试项：`<扩展名/MIME/解析/访问控制/执行链>`
- 最小输入：`<文件名 + MIME + 文件内容要点>`
- 结果：`<上传响应 + 访问验证 + 可复现次数>`
- 结论：`confirmed/suspected/not vulnerable`

- 约束：一次只改一个变量；发现高危证据立即停止并输出可复现步骤。
