# SQL 注入检测：多策略综合（Orchestrator）

## 目标
在不明确注入类型/数据库的情况下，先做低成本探测（基线、闭合、指纹），再按决策树选择并加载对应子技能（报错/UNION/布尔盲注/时间盲注），最终给出可复现的证据链与最小 PoC。

## 使用方式（关键）
- 本技能是"路由器"：当用户只说"测 SQL 注入 / 注入点不确定 / 想综合判断"时优先使用。
- 执行过程中按条件调用 `load_skill` 加载子技能正文（只加载需要的，避免上下文膨胀）：
  - `load_skill({"skill_key":"SQL注入-报错注入检测"})`
  - `load_skill({"skill_key":"SQL注入-UNION注入检测"})`
  - `load_skill({"skill_key":"SQL注入-布尔盲注检测"})`
  - `load_skill({"skill_key":"SQL注入-时间盲注检测"})`

## 全局约束（默认保守预算）
- 单参数最多 20 次请求（含基线/重试）；同一轮只改 1 个参数。
- 时间盲注在足以区分抖动的前提下选择合适延时与样本量，避免机械固定次数。
- 并发建议 1；若抖动大，增加重试而不是加大延时/并发。
- 一旦"确认"则停止进一步探测（避免无意义扫描）。

## 参数定位与隔离
- 优先挑选高风险参数：`id`、`uid`、`orderId`、`q`、`keyword`、`sort`、`filter`、`name` 等。
- 参数位置按优先级依次排查：Query → Body → Cookie → Header（仅改 1 个位置，避免多点联动）。
- 每轮只改一个参数，便于将响应差异与该参数建立因果关系。

## 统一证据记录（建议结构）
每一步都记录以下字段，便于复核与报告：
- 改动点：参数名、位置（query/body/cookie/header）、原值→新值
- 响应特征：status、重定向链、长度、关键字命中、稳定片段差异（可用 diff 工具对比）
- 耗时统计：每次耗时、均值/中位数、抖动范围
- 可复核性：采用足以排除随机噪音和业务波动的方式确认同一 payload 的差异具有稳定因果关系

## 工作流（决策树）

### Step 0：基线采集
1. 发送原始请求 2~3 次，得到"基线响应特征"和"基线耗时分布"。
2. 标记动态字段（时间戳、随机 ID）并在差异对比时忽略。

### Step 1：上下文/闭合探测（字符/数字/括号）
目标是快速判断"参数更像数字型还是字符串型，是否需要 `)` 闭合，以及注释风格是否可用"。

最小探测集（按需选择，避免全打）：
- 引号探测：在参数末尾追加 `'` 或 `"`，观察是否出现**新**的语法错误特征/响应突变（与基线对比）。
- 括号探测：在疑似函数/括号场景下追加 `)` / `))`，观察是否从"错误→正常"或"正常→错误"发生可复现翻转。
- 注释候选：若需要截断后续语句，优先 `-- `；MySQL 场景再尝试 `#`（都要结合闭合结果）。

输出：得到候选上下文集合（示例）：
- numeric / string
- quote: `'` / `"` / none
- paren_close: 0/1/2
- comment: `-- ` / `#` / `/*...*/`

### Step 2：DBMS 指纹（只覆盖 MySQL/PG/MSSQL）
在不做深度利用的前提下，通过"错误关键字/函数特征"做轻量打分：
- MySQL 线索：`You have an error in your SQL syntax`、`SQLSTATE`、`MySQL` 等
- PostgreSQL 线索：`syntax error at or near`、`PostgreSQL`、`SQLSTATE` 等
- MSSQL 线索：`Unclosed quotation mark`、`ODBC`、`SQL Server`、`Microsoft` 等

指纹结论必须标注置信度（high/medium/low）；若不确定，后续时间盲注按 Top1 选择函数，失败再切换 Top2（默认不超过 2 种 DB）。

### Step 3：选择策略并加载子技能
按以下优先级选择（越靠前越"高信息量/低请求数"）：

1. **疑似错误回显/语法错误可触发**
   - 触发条件：追加轻微闭合/引号后出现稳定错误特征（且基线不存在）
   - 动作：加载并按其方法论执行：`SQL注入-报错注入检测`

2. **疑似 SELECT 查询且页面可回显（列表/搜索/详情）**
   - 触发条件：响应中存在可见字段回显位；或 `ORDER BY` 探测出现列数相关错误线索
   - 动作：加载并执行：`SQL注入-UNION注入检测`

3. **内容差异稳定（适合 TRUE/FALSE 对比）**
   - 触发条件：在少量探测中，响应关键字/结构可稳定对比
   - 动作：加载并执行：`SQL注入-布尔盲注检测`

4. **内容差异不稳定但耗时可测（最后手段）**
   - 触发条件：内容随机/模板化导致布尔差异不可靠，但基线耗时抖动可控
   - 动作：按指纹 Top1 DB 选择时间函数，加载并执行：`SQL注入-时间盲注检测`

### Step 4：确认标准（防误报）
- **强证据（满足其一且能说明具体可利用效果时可确认）**
  - 稳定的数据库错误回显且只由 payload 引发
  - 稳定的 UNION 回显（可控标记出现/消失）且可复现
- **弱证据（需两类同时满足才确认）**
  - 布尔盲注 TRUE/FALSE 差异可复现 + 另一项辅助证据（轻量错误线索/指纹一致/多次统计显著）
  - 时间盲注统计显著（差异远大于抖动）+ 指纹/闭合推断一致

若只出现单次异常、单纯报错、轻微差异或尚未说明具体可利用效果，结论降级为"可疑（suspected）"，并给出下一步验证建议。

## 闭环验证要求（必须遵守）
- 所有结论必须形成"前置条件/输入 -> 系统处理 -> 实际效果/危害 -> 可复核证据"的完整证据链，重点证明风险真实生效，而不是只看响应。
- 仅凭上传成功、请求成功、状态码变化、单次报错、配置落盘、前端提示、登录跳转、返回文案等中间信号，不得直接判定 `confirmed`。
- 若只有中间信号、缺少真实效果/危害验证，结论最多为 `suspected`。
- 对写操作、上传、配置变更、认证命中、权限变更、解析触发等场景，必须补做访问、回读、触发、解析、执行或副作用验证，证明风险真正落地并说明具体危害。
- 无法闭环时，必须明确缺失环节与下一步验证建议，不得跳过验证直接下结论。

## 实际效果验证方向（必须说明可利用能力）
- UNION：可控标记或数据库字段被真实回显，证明存在数据泄露能力。
- 报错注入：错误信息真实暴露了数据库类型、SQL 结构、字段/函数结果或其他可利用信息，而不是普通 500。
- 布尔盲注：可稳定控制查询真假分支，影响页面数据存在性、业务分支或结果集判断。
- 时间盲注：可稳定控制延时侧信道，证明后续能够进行条件推断，而不是普通网络抖动。
- 若只能证明"有报错/有差异/有延时"，但说不清可造成什么能力或泄露什么信息，不能直接给 `confirmed`。

## 最小 PoC 输出模板
- 目标：`<URL/接口>`
- 参数：`<name> (<位置>)`
- 上下文推断：`numeric|string + quote + paren_close + comment`
- DBMS 推断：`MySQL|PostgreSQL|MSSQL (confidence=...)`
- PoC（最小复现）：给出 1 组对照（TRUE/FALSE 或 延时/不延时）
- 证据：差异点（diff 结果要点/错误关键字/耗时统计）

## AI-Fuzz 提示
- 目标：在"闭合正确 + 请求预算受控"的前提下，生成少量高信息量探测序列（先错误/指纹，再 UNION/布尔/时间）。
- 约束：每个参数单轮只改 1 处；设置熔断与最大请求数；发现强证据立即停止。
