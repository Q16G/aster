# JWT 检测：弱密钥与信息泄露

## 目标
检测 JWT 是否存在以下风险：
- payload 直接暴露敏感信息（账号、权限、密钥片段等）
- HMAC 签名密钥过弱（可被字典验证命中）

## 适用场景
- 基于 Bearer Token 的 API 鉴权系统。
- 登录认证、网关转发、单点登录（SSO）场景。
- 需要审查 token 内容最小暴露与签名安全性的接口。

## 工具使用
- `jwt_decode`：先对 token 做无签名校验解码，识别 `alg`、`exp`、敏感 claims。
- `jwt_crack`：仅在 `alg` 为 `HS256/HS384/HS512` 时执行受控弱密钥校验。
  - 支持 `key` 指定密钥定向解密；
  - 支持 `use_builtin_wordlist=true` 启用内置 `jwt.secrets.txt` 批量解密。

## 前置条件与安全边界
- 仅分析授权获取的 token，禁止采集或传播无授权敏感凭证。
- 弱密钥验证仅限小规模受控字典，不进行高强度暴力尝试。
- 非 HMAC 算法（如 RS256/ES256）不做"弱密钥爆破"结论。
- 若命中高风险，立即停止扩展尝试并进入修复建议阶段。

## 方法论
1. **基线采集**：提取请求中的 `Authorization` header / cookie / body 中的 JWT。
2. **解码分析**：调用 `jwt_decode`，记录算法、过期字段、敏感 claims。
3. **弱密钥验证（条件执行）**：若为 HMAC 算法，优先尝试已知 `key`；未知时调用 `jwt_crack` 并按需开启内置字典批量验证。
4. **风险判定**：
   - 命中弱密钥：高危
   - payload 含敏感字段且无必要：中危
   - 无明显问题：信息项

## 闭环验证要求（必须遵守）
- 所有结论必须形成"前置条件/输入 -> 系统处理 -> 实际效果/危害 -> 可复核证据"的完整证据链，重点证明风险真实生效，而不是只看响应。
- 仅凭上传成功、请求成功、状态码变化、单次报错、配置落盘、前端提示、登录跳转、返回文案等中间信号，不得直接判定 `confirmed`。
- 若只有中间信号、缺少真实效果/危害验证，结论最多为 `suspected`。
- 对写操作、上传、配置变更、认证命中、权限变更、解析触发等场景，必须补做访问、回读、触发、解析、执行或副作用验证，证明风险真正落地并说明具体危害。
- 无法闭环时，必须明确缺失环节与下一步验证建议，不得跳过验证直接下结论。

## 实际效果验证方向（至少证明一类）
- 弱密钥命中后，能够最小化伪造或篡改 token，并被受保护接口接受。
- claims 泄露真实暴露了不必要的敏感身份、权限或业务数据，且接收方范围不应看到这些内容。
- 若只有弱密钥命中结果或 claims 可见，但未证明实际可利用能力或暴露影响，不能直接给 `confirmed`。

## 判定标准

| 现象 | 判定 |
|------|------|
| `jwt_crack` 明确命中 HMAC 签名密钥，且可据此最小化伪造/篡改 token 并访问受保护资源或改变权限语义 | confirmed |
| 命中弱密钥但尚未验证受保护资源接受，或 claims 暴露不必要敏感信息但影响范围未完全确认 | suspected |
| 算法与内容均无明显问题，且未出现弱密钥命中证据 | not vulnerable |

## 误报控制
- 仅在 token 格式合法时进行结论判定。
- 弱密钥结论必须来自 `jwt_crack` 明确命中；若要判 `confirmed`，还需证明该密钥可带来真实 token 篡改/伪造能力。
- 非 HMAC 算法（如 RS256）不做弱密钥爆破结论。

## 最小 PoC 输出模板
- 目标接口：`<METHOD> <URL>`
- Token 来源：`Authorization/Cookie/Body`
- 解码结果：`alg=<...>, exp=<...>, claims=<关键字段>`
- 弱密钥验证：
  - `jwt_crack` 输入：`<模式/字典>`
  - 输出：`<命中/未命中>`
- 结论：`confirmed | suspected | not vulnerable`

## 修复建议
- HMAC 模式使用高强度随机密钥并定期轮换，避免默认/弱口令密钥。
- 优先采用非对称签名（如 RS256/ES256）并严格管理私钥。
- claims 仅保留最小必要信息，避免放入敏感身份与业务数据。
- 严格校验 `exp/nbf/iss/aud`，并在服务端实现 token 撤销与失效策略。
