# IDOR 检测（水平越权）

## 目标
确认账号 A 是否可通过替换资源标识访问或操作账号 B 的资源。

## 适用场景
- 用户资料、订单、文件、工单、租户资源等"按资源归属控制"的接口。
- URL、Query、Body、Header 中存在可枚举资源标识（ID/UUID/路径片段）。

## 前置条件与安全边界
- 准备两组不同归属资源：`A 自有资源` 与 `B 自有资源`。
- 仅在授权环境测试，不对真实业务数据做破坏性修改。
- 单资源类型默认最多 8 次请求（A 基线、B 对照、复验）。
- 单轮只替换一个资源标识，避免多变量干扰。

## 检测步骤
1. 识别资源标识符位置（路径、查询参数、请求体、批量 ID 列表）。
2. 账号 A 访问 A 资源建立基线，记录关键归属字段（owner_id、tenant_id）。
3. 账号 A 保持其凭证不变，仅将资源标识替换为 B 资源并重放请求。
4. 对写操作执行回读验证，确认 B 资源是否被实际修改。
5. 对可疑结果复验，排除缓存、异步延迟和数据脱敏导致的误判。

## 闭环验证要求（必须遵守）
- 所有结论必须形成"前置条件/输入 -> 系统处理 -> 实际效果/危害 -> 可复核证据"的完整证据链，重点证明风险真实生效，而不是只看响应。
- 仅凭上传成功、请求成功、状态码变化、单次报错、配置落盘、前端提示、登录跳转、返回文案等中间信号，不得直接判定 `confirmed`。
- 若只有中间信号、缺少真实效果/危害验证，结论最多为 `suspected`。
- 对写操作、上传、配置变更、认证命中、权限变更、解析触发等场景，必须补做访问、回读、触发、解析、执行或副作用验证，证明风险真正落地并说明具体危害。
- 无法闭环时，必须明确缺失环节与下一步验证建议，不得跳过验证直接下结论。

## 实际效果验证方向（至少证明一类）
- 跨主体读取：A 读取到 B 的真实资源内容、字段或文件。
- 跨主体修改：A 对 B 的资源执行写操作，并通过回读/状态变化证明影响真实生效。
- 若只有 200、空壳响应或 ID 可猜测，但未证明跨主体真实访问效果，不能给 `confirmed`。

## 判定标准

| 现象 | 判定 |
|------|------|
| 账号 A 成功读取或修改账号 B 资源，并通过真实回读或状态变化证明跨主体影响成立 | confirmed |
| 行为异常（如 200 空壳）但尚未证明跨主体真实读取或写入效果 | suspected |
| 账号 A 无法访问 B 资源（403/404 且无泄露） | not vulnerable |

## 误报控制
- "数据不同"不等于越权成功，必须校验归属字段确为 B 主体。
- 对返回 `404` 的场景，需结合日志或对照请求判断是否做了"权限后隐藏"。
- 对列表接口需验证是否混入他人记录，而非单条接口限定判断。
- 写操作必须以实际持久化结果为依据，不以接口即时回包单独定论。

## 最小 PoC 输出模板
- 目标接口：`<METHOD> <URL>`
- 主体与资源：
  - `User-A` 拥有 `Resource-A`
  - `User-B` 拥有 `Resource-B`
- 对照请求：
  - `Req-A(User-A -> Resource-A) -> <基线结果>`
  - `Req-B(User-A -> Resource-B) -> <结果>`
- 关键证据：`<owner_id/tenant_id/写后读结果>`
- 结论：`confirmed | suspected | not vulnerable`

## 修复建议
- 资源查询必须同时绑定"资源 ID + 当前主体 ID/租户 ID"。
- 禁止仅依赖前端传入的 `user_id/tenant_id` 决策权限。
- 对批量接口逐项做归属校验，不因单项合法而整体放行。
- 为关键资源接口补充 IDOR 回归测试（跨用户/跨租户场景）。
