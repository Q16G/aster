# 反滥用检测：邮箱/短信轰炸（Orchestrator）

## 目标
在不明确发送限制、冷却时间、人机挑战和多维频率限制策略时，先做基线与成功判据确认，再按证据选择子 skill 深挖，最终输出低样本、可复现、可回归的最小 PoC 与修复建议。

## 使用方式（关键）
- 本技能是"路由器"：当用户只说"测邮箱轰炸 / 短信轰炸 / 验证码轰炸 / 发送频率限制"时优先使用。
- 执行时按条件调用 `load_skill` 读取对应子技能正文（只加载需要的，避免上下文膨胀）：
  - `load_skill({"skill_key":"通知滥用-单目标连续发送检测"})`
  - `load_skill({"skill_key":"通知滥用-冷却时间与时间窗口检测"})`
  - `load_skill({"skill_key":"通知滥用-多维频率限制绕过检测"})`
  - `load_skill({"skill_key":"通知滥用-验证码与人机挑战检测"})`
  - `load_skill({"skill_key":"通知滥用-多目标切换与资源消耗检测"})`

## 前置条件与安全边界
- 仅对自有测试手机号、测试邮箱和授权环境执行，不对真实用户实施骚扰。
- 单目标默认最多 12 次请求；单接口总预算默认最多 30 次请求；目标数量默认不超过 3 个。
- 默认串行请求，间隔 0.5~1 秒；不能只看"业务成功响应"，至少要结合发送流水/消息 ID、额度变化、队列状态、冷却状态或自有测试目标收件结果之一验证真实发送效果。
- 禁止为了验证而扩大攻击面；一旦确认缺少有效防护，立即停止并进入证据固化。

## 参数与面识别
优先识别以下要素：
- 目标参数：`phone`、`mobile`、`tel`、`email`、`account`、`receiver`
- 业务场景参数：`scene`、`bizType`、`action`、`type`
- 反滥用相关参数：`captcha`、`ticket`、`nonce`、`device_id`、`session`、`uid`
- 来源识别链：`X-Forwarded-For`、`X-Real-IP`、`Forwarded`、网关注入的真实源 IP 头
- 成功判据：业务码、`success=true`、剩余冷却秒数、提示文案、风控标记

## 统一证据记录（建议）
每轮测试记录：
- 输入变化：目标值（脱敏）、场景、是否登录、Cookie/Session、挑战参数
- 输出特征：HTTP 状态、业务码、消息文案、冷却秒数、是否返回挑战字段
- 时间特征：请求时间点、间隔、等待后是否恢复发送
- 可复现性：同一条件至少 2 次一致

命中"邮箱/短信轰炸"后，默认按以下顺序执行：
1. **基线与成功判据确认**

2. **单目标连续发送检测**
   - 加载 `通知滥用-单目标连续发送检测`。
3. **冷却时间与时间窗口检测**
   - 加载 `通知滥用-冷却时间与时间窗口检测`。
4. **验证码与人机挑战检测**
   - 加载 `通知滥用-验证码与人机挑战检测`。
5. **多维频率限制绕过检测**
   - 加载 `通知滥用-多维频率限制绕过检测`。
6. **多目标切换与资源消耗检测**
   - 加载 `通知滥用-多目标切换与资源消耗检测`。
7. **复测与结论固化**

## 测试项总清单（命中后应覆盖）
- 单目标连续请求是否可持续成功
- 是否存在服务端冷却时间与时间窗口计数
- 返回的等待秒数/剩余额度是否真实生效
- 阈值后是否出现验证码、滑块、人机挑战或二次校验
- 风控是否只在前端展示，服务端是否真正强制校验
- 频率限制是否覆盖账号、目标、IP、设备、Session、业务场景等维度
- 是否错误信任 `X-Forwarded-For`、`X-Real-IP`、`Forwarded` 等可伪造来源头
- 目标标准化是否覆盖前后空格、Tab、全角空格、URL 编码空白（如 `%20`、`%09`）
- 手机号标准化是否覆盖区号前缀、空格、横杠、括号等格式变体
- 邮箱标准化是否覆盖大小写、前后空白、业务允许范围内的别名/标签变体
- 低成本切换上下文后是否可重新发送
- 2~3 个测试目标轮换时是否存在全局限额与总量控制
- 响应是否暴露剩余额度/风控细节，便于自动化轰炸调优
- 证据是否可在低样本、低干扰条件下稳定复现

## 常见低成本绕过向量（优先小样本验证）
- **来源识别链**：在确认系统依赖源 IP 维度时，小样本比较 `X-Forwarded-For`、`X-Real-IP`、`Forwarded` 是否影响额度判断。
- **空白字符**：目标参数两端的普通空格、Tab、全角空格，以及 URL 编码空白（如 `%20`、`%09`）。
- **手机号格式化**：`+86`、`0086`、空格、横杠、括号等仅影响展示、不应重置频控的格式差异。
- **邮箱归一化**：大小写差异、前后空白，以及业务明确支持时的别名/标签变体。
- **上下文切换**：Session、设备标识、匿名/登录态、业务场景等单一维度变化。

## 决策树

### Step 0：基线与成功判据确认
1. 先用 1 个测试目标发送 1~2 次，确认成功业务码、失败业务码和提示字段。
2. 区分"HTTP 200 但业务失败"和"真正发送成功"，避免把统一响应误判为成功。

### Step 1：选择子技能
1. **同一目标连续发送持续成功**
   - 线索：短时间重复发送都返回成功，未提示等待/挑战
   - 动作：加载 `通知滥用-单目标连续发送检测`
2. **响应出现等待秒数/频率提示，但怀疑只做了前端或时间窗不生效**
   - 线索：返回"请 60 秒后再试"但服务端仍可继续成功
   - 动作：加载 `通知滥用-冷却时间与时间窗口检测`
3. **理论上应出现图形验证码/滑块/二次校验，但实际未出现或未校验**
   - 线索：阈值后仍无挑战，或不带挑战凭证也成功
   - 动作：加载 `通知滥用-验证码与人机挑战检测`
4. **怀疑限制仅绑定单一维度，切换上下文即可继续发送**
   - 线索：换 Session/设备标识/登录态后成功率恢复，或仅改 `XFF` / 目标参数空白与格式变体后额度被重置
   - 动作：加载 `通知滥用-多维频率限制绕过检测`
5. **单目标有局部限制，但轮换多个目标仍可稳定消耗渠道资源**
   - 线索：每个目标各能成功多次，缺少接口级/账号级总量控制
   - 动作：加载 `通知滥用-多目标切换与资源消耗检测`

### Step 2：确认标准
- `confirmed`：满足以下任一项，且能证明真实发送影响或风控失效：
  - 单目标在预算内可持续形成真实发送效果/额度消耗，且未触发有效冷却或挑战
  - 服务端返回的冷却/时间窗口规则可被低成本绕过，且绕过后真实发送能力恢复
  - 阈值后的人机挑战仅停留在前端，服务端未强制校验，导致真实发送继续发生
  - 仅通过 `XFF`、空格、格式化/标准化变体就能重置发送额度或继续成功发送
  - 切换低成本上下文后真实发送额度被重置，证明限流维度不足
  - 2~3 个测试目标轮换时缺少接口级总量控制，仍可稳定形成真实发送或额度消耗
- `suspected`：存在限制缺口迹象，但仅有接口成功/文案变化，尚未证明真实发送影响，或环境存在 mock/灰度干扰。
- `not vulnerable`：在预算内已触发并验证有效的服务端限流、冷却和挑战机制。

## 闭环验证要求（必须遵守）
- 所有结论必须形成"前置条件/输入 -> 系统处理 -> 实际效果/危害 -> 可复核证据"的完整证据链，重点证明风险真实生效，而不是只看响应。
- 仅凭上传成功、请求成功、状态码变化、单次报错、配置落盘、前端提示、登录跳转、返回文案等中间信号，不得直接判定 `confirmed`。
- 若只有中间信号、缺少真实效果/危害验证，结论最多为 `suspected`。
- 对写操作、上传、配置变更、认证命中、权限变更、解析触发等场景，必须补做访问、回读、触发、解析、执行或副作用验证，证明风险真正落地并说明具体危害。
- 无法闭环时，必须明确缺失环节与下一步验证建议，不得跳过验证直接下结论。

## 实际效果验证方向（必须说明真实发送影响）
- 服务端真实创建了发送任务、消息流水、队列记录或额度消耗。
- 自有测试手机号/邮箱真实收到通知，或后台发送记录/审计日志可证明已发。
- 限流/挑战绕过后，真实发送能力被恢复，而不是只有接口返回成功。
- 若只能证明"接口返回成功"，但未证明真实发送、额度消耗或风控失效，不能给 `confirmed`。

- 接口：`<METHOD> <URL>`

- 业务场景：`<register/login/reset/notify>`
