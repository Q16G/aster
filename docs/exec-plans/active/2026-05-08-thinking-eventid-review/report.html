Thinking EventID 方案评审

评审对象：飞书文档 `W6d2dfibVo282exgL6FcErAFnRW` ｜ 结论：方案方向正确，但“根因叙述”与“session reset 边界”还需要再对齐一次

Background / 任务背景

TUI 的 thinking 分块问题，当前方案值不值得继续推进？

文档提出的问题是：在 step_summary / final_answer 阶段，reasoning token 会被拆成多个 ThinkingPart，在 TUI 中表现为逐行堆叠。给出的方案是在事件层引入稳定的 EventID，让 TUI 能按 thinking session 做聚合。

我对照了飞书文档、当前未提交改动和相关 phase 的真实事件时序。结果是：这套设计思路本身是合理的，也已经以较小代价接进了 runtime 和 TUI；但“问题是否真的是被 StateChange 打断”这件事，当前代码证据还不够闭环。

评审范围：internal/react/emitter.go、internal/tui/chat.go、internal/tui/display_part.go、internal/tui/event_handlers.go、internal/tui/chat_test.go
验证结果：go build ./... 通过；go test ./internal/tui/... ./internal/react/... 通过
判断：实现方向正确，证据链还差一条“真实事件序列回归测试”
What / 做了什么
  1. 读取飞书文档的问题、根因、参考实现、改动方案和验证方式。
  2. 逐个核对 emitter、TUI 聚合逻辑、event handler 和测试用例。
  3. 回溯 step_summary / final_answer phase 的真实事件产生路径。
  4. 执行编译与定向测试，确认当前补丁没有破坏主路径。
Result / 实现效果

结论：继续推进这套方案是合理的，但建议先把“哪些事件应该 reset thinking session”再缩窄一轮，并补一条基于真实事件流的回归测试。否则我们只能证明“ChatModel 会按 EventID 合并”，还不能证明“真实问题场景一定会复用同一个 EventID”。

如果实际问题真的是“同一段 reasoning 被 StateChange 打断”，那当前对 EmitStateChange() 的 reset 策略反而会保留这次分裂，而不是消除它。
Todo / 待优化
  • 用真实复现序列补一个 handler/phase 级测试，而不只是在 ChatModel 里手工调 FlushThinking()。
  • 根据复现结果重新确认 ResetThinkSessionID() 的白名单/黑名单事件集合。

关键矛盾点就在右侧这条线：如果复现中的“打断事件”本身也会触发 ResetThinkSessionID()，那么后续 delta 就不再属于同一 EventID，merge 逻辑也就没有机会生效。

How / 怎么做的

主要判断依据

方向合理：internal/react/emitter.go 给 thinking 流补了稳定 EventID，internal/tui/chat.go 用最小侵入方式补了按 ID 合并，设计上和已有 ToolPart.CallID 的思路一致。

证据不闭环：internal/react/structured_output.go 的 stream handler 只发 EmitThink()；step_summary / final_answer 的 EmitStateChange() 主要在流开始前和结束后。这意味着文档里“被 StateChange 打断”的说法需要拿真实 trace 再证一次。

风险点：当前实现在 internal/react/emitter.go 里对 EmitStateChange() 也做了 ResetThinkSessionID()。如果真实复现里就是这个事件造成 split，那么补丁只能保证“下一段 thinking 新开 session”，无法把它和前一段重新合并。

验证面：internal/tui/chat_test.go 已经覆盖同 ID 合并、不同 ID 分离和 session 切换，但都发生在 ChatModel 内部，还没有一条“真实 runtime 事件顺序”的回归测试。
