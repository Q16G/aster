ASTER：在 AI 时代深耕安全分析的 Agent 框架

深度技术解析 | Agent 架构 | 安全实战

一、为什么我们要造一个安全分析 Agent

核心观点：安全工作的本质是"在不确定性中做判断"——无论是代码审计、渗透测试还是应急响应，安全工程师的核心能力不是执行固定流程，而是根据上下文灵活决策。这恰恰是 Agent 最擅长的事情。

安全工作的三个场景

安全工程师的工作远不止"扫描代码找漏洞"。根据场景不同，安全工作至少涵盖三大领域：

[ 图：安全工作三大领域 — 代码审计 / 渗透测试 / 应急响应 ]

这三个领域有一个共同特征：都不是"运行一个工具就能得到答案"的简单任务。它们都需要安全工程师根据项目的具体情况制定策略、选择工具、解读结果、调整方向——这是一个持续的推理-行动-观察循环。

传统工具链的困境

❌ 工具孤岛

SAST、DAST、SCA 各自为战。Semgrep 的扫描结果无法自动传递给数据流引擎做验证，渗透工具发现的线索无法反馈到代码层面做确认。工程师需要手动在工具之间搬运上下文。

❌ 缺乏判断力

传统工具只会"报告发现"，不会"做出判断"。Semgrep 告诉你"这里有 SQL 拼接"，但不会告诉你"这个参数是否真的来自用户输入"。判断的责任完全压在人身上。

❌ 无法编排

安全分析有天然的顺序逻辑：先了解攻击面，再针对性扫描，再深入验证，再评估影响。但没有一个工具能编排这个完整流程——工程师就是那个"人肉编排器"。

为什么是 Agent

大语言模型（LLM）具备语义理解和推理能力，可以阅读代码、分析配置、理解业务逻辑。但 LLM 本身只是"大脑"——它需要"手脚"（工具调用）来执行具体操作，需要"工作流"（规划与调度）来组织多步骤任务。

Agent = LLM + 工具 + 工作流。它不是一个简单的"AI 问答"，而是一个能够自主规划、调用工具、观察结果、调整策略的执行体。对于安全分析这种需要多步骤、多工具协同、有反馈回路的领域，Agent 是天然的解决方案。

Agent 的核心循环

📋 规划 ➔ ⚡ 行动 ➔ 👁 观察 ➔ 🔄 反思 ➔ 🔁

为什么自建而非复用

现有 Agent 框架（LangChain、AutoGen 等）面向通用场景设计，而安全领域有三个特殊需求：

1. 领域知识编排：安全审计需要先 SAST 扫描建立候选集，再数据流追踪验证可达性，再认证复核评估影响——这种编排逻辑需要内化到框架中，而非每次都依赖 LLM 自发行为。

2. 超长任务的上下文管理：一次完整的安全审计可能持续数百轮工具调用，需要精确的 token 预算管理和历史压缩策略。

3. 精确的权限控制：安全工具涉及 bash 命令执行、网络请求等敏感操作，需要比通用框架更精细的权限审查机制。

于是我们从零构建了 ASTER——一个专为安全领域设计的 Agent 框架。

二、多智能体架构的迷思：为什么我们选择了 Profile 而非 Multi-Agent

核心论断：Agent 的个性化是知识层面的，不是运行时层面的。多智能体架构的真正价值不在于"多个独立智能体互相对话"，而在于"同一个运行时能以不同的知识和工具集执行不同的任务"。

Multi-Agent 热潮

2024 至 2025 年，Multi-Agent 成为 AI 应用领域最热门的叙事之一。学术界和工业界涌现了大量多智能体框架与综述：

框架 来源 核心贡献
AutoGenMicrosoft Research构建多 Agent 对话框架
MetaGPTICLR 2024引入 SOP 结构化多智能体协作
ChatDevACL 2024多个 Agent 协作开发软件
CrewAI开源社区提出「AI 员工团队」概念

这些框架的核心主张高度一致：将复杂任务拆解给多个专业化的 Agent，通过协作完成单个 Agent 无法胜任的工作。

这个叙事听起来非常有吸引力——就像真实的团队协作一样，让"安全专家 Agent"负责漏洞扫描，"代码专家 Agent"负责代码理解，"报告专家 Agent"负责输出整理。但在实际的代码安全分析场景中，这个架构存在根本性的问题。

隐含假设的崩塌

Multi-Agent 架构有一个隐含假设：智能体之间的通信和协调成本，低于单个智能体处理全部任务的认知负担。在代码安全分析中，这个假设不成立。原因有三：

🔗 上下文连续性

一次完整的安全审计需要从项目结构理解、攻击面枚举，到具体漏洞确认、影响评估的完整链路。把它拆成多个 Agent，意味着上下文碎片化。AgentScope 的实验验证了这一点：10 轮 Agent 间对话后，信息保真度降至约 60-70%。

⚡ 反馈回路的即时性

安全分析是高度迭代的：发现一个 SQL 注入候选后，需要立即做数据流追踪验证；追踪结果可能暴露新的攻击面，又需要回头检查认证授权。多 Agent 通信引入的序列化开销和信息编解码损耗是致命的。研究证实，Agent 数量增加后通信开销呈超线性增长，还会出现「社会懒惰」现象。

🔧 工具集的高度重叠

无论是"扫描 Agent"还是"分析 Agent"，它们都需要读取文件、搜索代码、执行命令。工具集完全重叠，真正的区别只在于知识。SWE-agent 得出了同样结论：工具接口设计比增加 Agent 数量更重要。

安全分析：单智能体的主场

这不仅是理论推导——安全领域的实证研究已经给出了答案：

🤝 多 Agent 的价值

多个 reviewer Agent 从不同角度审查代码能发现更多问题——多智能体在「多角度审查」上确实有价值。

✅ ASTER 的替代方案

同一个 Agent 串行执行多个 Skill，每个 Skill 聚焦不同的审计维度，就能获得同等的覆盖广度，同时保持上下文的完整性。

从理论推导到实证验证，结论指向同一个方向：安全分析需要的不是「多个运行时互相对话」，而是「一个运行时以不同的知识工作」。

三、从 LangChain 到 Claude Code：ReAct 实现的三代演进与未解之题

核心观点：每一代 ReAct 框架都解决了上一代的部分问题，但也引入了新的局限。从"过度抽象"到"过度自治"再到"通用但不专"，Agent 框架的演进史本身就是一部对"自主性与可控性"平衡的探索史。

三代 ReAct 框架对比

代次 代表 核心贡献 关键问题
第一代LangChain建立 Agent = LLM + Tools 范式过度抽象，状态管理弱
第二代AutoGPT赋予 Agent 自主规划执行能力过度自治，缺乏 guard rails
第三代Claude Code工具调用与对话深度集成，TUI 体验优秀面向通用编码，非安全领域

第一代的教训：抽象不等于能力

调试困难 + 状态缺失

Chain 嵌套导致逐层 debug；无检查点机制，中断则全部进度丢失

缺乏反思机制

执行"一步一步"，完成后直接进入下一个，没有"回头看"

第二代的教训：自治需要边界

AutoGPT 走向了另一个极端：给 Agent 几乎无限的自主权，让它自己决定做什么、怎么做。结果是 Agent 经常在无关的方向上耗费大量 token，或者陷入重复循环无法跳出。

第三代：Claude Code 的启示与不足

✅ Claude Code 的设计哲学

• 工具调用不是附加功能，而是核心能力——模型原生支持 tool use
• 流式输出让用户实时看到 Agent 的思考与执行过程
• 内置工具集经过精心筛选，覆盖文件读写、搜索、终端等高频操作

安全领域的差距

• 没有安全分析所需的领域知识编排
• 没有显式的多阶段执行流程
• 没有 Profile 切换机制，无法按场景加载不同策略

演进规律：每一代产品都在修复上一代最大的痛点，同时又不自觉地制造新的问题。这就是我们构建下一代工具的起点。

四、ASTER 的 Agent 架构

致敬：ASTER 的很多设计灵感来自 Claude Code——工具精选、流式交互、TUI 体验。我们在此基础上，增加了安全领域所需的显式执行阶段、领域 Skill 系统和声明式 Agent Profile。

整体架构

ASTER 的架构分三层：

上层 - Profile 适配层：让不同角色的 Agent 共享同一个运行时——同一套代码既能做代码审计，也能做渗透测试，区别仅在一份 YAML 配置。

中层 - ReAct 引擎：提供统一的四阶段执行循环。

底层 - 能力层：包含内置工具、51 个安全 Skill 和 MCP 协议扩展。

四阶段执行：Agent 怎样完成一个任务

大多数 Agent 框架用一个大循环处理一切——模型自行决定什么时候规划、什么时候执行、什么时候收工。ASTER 把执行过程显式分为四个阶段：

🎯 ➔ Plan
任务规划 ➔ Step
执行步骤 ↔ Replan
动态调整 ➔ FinalAnswer
最终评估 ➔ ✅

阶段 职责 关键机制
Plan分析任务全貌，选择 Skill，制定执行顺序先全局规划，不会"上来就干"
Step逐步执行计划每步独立的思考-行动-观察循环
StepReplan每步完成后评估新发现独立 LLM 调用，可主动调整
FinalAnswer所有步骤完成后独立评估发现遗漏可回流 Plan 继续

解决核心问题：「假完成」
单循环 Agent 倾向于做完预定步骤就宣布完成，即使实际上遗漏了关键维度。独立的 FinalAnswer 评估打断了这种惯性。遇到高风险操作时，Agent 会触发 Human-in-the-loop 中断等待确认，完整的运行时状态被持久化，确认后从断点恢复。

Profile 系统：一个引擎，多种角色

ASTER 不需要为每个安全场景开发独立的 Agent。所有角色共享同一个 ReAct 引擎，区别仅在于一份 YAML 配置中声明的角色定义、执行指令、Skill 集合和安全策略。

自定义 Profile：用户可以在 ~/.aster/agents/ 目录下创建自己的 YAML Profile——定义角色、选择 Skill、配置策略，无需修改任何源代码。这让 ASTER 成为一个安全 Agent 开发框架，而不仅仅是一个固定用途的工具。

Skill 系统：领域知识的结构化表达

ASTER 内嵌了 51 个安全 Skill（渗透测试 27、代码审计 16、主机防护 5、通用 2、漏洞复现 1），每个 Skill 是一份 Markdown 文档，包含完整的决策树、闭环验证要求和误报缓解策略。Skill 通过 load_skills 按需动态加载到上下文窗口。

五、Skill 系统设计：把安全专家的知识固化成可执行的 Checklist

核心观点：Skill 本身是静态的 checklist——安全专家的完整决策树、验证要求和误报策略被编码成结构化文档，不随模型变化而变化。模型负责动态的部分：根据实际上下文判断哪些检查项可以跳过。静态 checklist 保证覆盖率下限，模型能力决定分析深度上限。

5.1 设计哲学：Checklist 兜底基础面，模型泛化其余面

Skill 系统要解决一个核心问题：AI 不擅长做加法。让模型自行决定「应该检测哪些安全维度」，它一定会遗漏——SQL 注入有 6 种数据库变体、文件上传有十几种绕过手法，靠模型临场枚举是不现实的。

❌ 加法思路（从空白枚举）

模型自行决定「应该查什么」，遗漏不可避免——开放性枚举没有终止条件

✅ 减法思路（从完整清单裁剪）

给模型完整 checklist，由它判断「什么可以不做」——事实判断，可靠

设计思路：把枚举变成裁剪
给模型一个完整的 checklist 作为起点，让它做减法而不是加法。「目标系统没有文件上传入口」是有客观依据的事实判断，不是开放性猜测。

核心洞察

Checklist 保证覆盖率下限，模型能力决定分析深度上限。
Checklist 给模型一个完整起点——模型做减法远比做加法可靠。

这个分工体现在两种执行模式中：

📋 全量模式（默认）

用户给出宽泛目标时，加载全部 checklist 项。模型逐项判断是否适用——没有明确否定证据就执行，同时自由补充 checklist 之外的发现。

🎯 聚焦模式

用户明确指定测试方向时，只加载对应 Skill 的 checklist。精力集中在少数维度上实现更高深度。

个安全 Skill

我们生活在一个 AI 工具泛滥的时代。每天都有新的"AI 驱动的安全工具"诞生，它们中的大多数本质上是一层薄薄的 API 封装——把代码丢给 LLM，让它输出一份看起来专业的报告。这种方式能应付演示，但经不起实战检验。

ASTER 选择了一条不同的路：不是用 AI 快速做出一个东西，而是用 AI 深入解决一个真实的问题。安全分析不是一个可以被"快速解决"的问题——它需要对漏洞模式的系统性理解、对数据流传播的精确追踪、对业务语义的深入分析。

大部分开发者在 AI 时代追求的是速度——更快地生成代码、更快地上线产品。我们选择追求深度——更深入地理解问题、更精确地解决问题。我们相信，当 AI 生成代码的成本趋近于零时，真正有价值的将不再是"谁写得更快"，而是"谁理解得更深"。

致敬 Claude Code

ASTER 的很多设计灵感直接来自 Claude Code——工具精选而非堆砌、流式输出作为基本权利、TUI 作为交互范式。Claude Code 证明了一件重要的事：好的 Agent 体验不需要复杂的架构，需要的是对细节的极致关注。

但 ASTER 的目标不是成为另一个 Claude Code。它要证明的是：Agent 框架不必是通用的——当你深入理解一个特定领域的需求，并为之量身定制时，你能获得通用框架无法企及的专业深度。
